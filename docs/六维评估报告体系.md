# 六维评估报告体系

本项目的评估报告采用“六维证据驱动”结构：总览只负责给结论，详情必须能追溯到页面、状态、交互用例、视觉区域、问题和证据产物。

## 报告产物

评估产物写入当前 project：

```text
projects/{target-id}/evaluation/latest/
├── summary.json          # 六维汇总，供 CI 或自动化读取
├── details.json          # 页面、状态、维度、问题、证据完整详情
├── artifacts-index.json  # 截图、diff、交互失败证据等产物索引
├── report.md             # 六维 Markdown 报告
├── index.html            # 六维 Dashboard 报告
├── captures/
└── assets/
```

## 报告结构

`report.md` 与 `index.html` 使用同一份 `details.json` 渲染，结构保持一致：

1. 总览：总分、结论、门禁、最低维度、页面/状态/高危问题数量。
2. 页面 × 六维矩阵：每个页面的总分和六维得分。
3. 状态矩阵：状态是否 required、门禁是否通过、对应六维表现。
4. 问题中心：每个问题都有维度、严重级别、影响分、置信度和修复建议。
5. 六维详情：每个维度包含子项、权重、分数、证据和修复建议。
6. 证据中心：按维度归档截图、diff、交互失败截图和采集数据。

## 硬门禁

硬门禁失败时报告仍会生成，但结论为 `failed`，总分只用于定位问题，不作为通过依据。硬门禁包括：

- 原站 Phase 3 基线可信。
- 复刻页面可运行。
- required 状态可达。
- 禁止截图伪页面。
- 核心交互无阻断错误。
- 评估产物可复现。

## 六维指标

| 维度 | 权重 | 报告关注点 |
| --- | ---: | --- |
| 功能正确性 | 25% | required 状态、核心用例、输出结果、边界/异常态 |
| 交互流程一致性 | 20% | 点击、输入、Enter、焦点、反馈、分页和状态切换 |
| 视觉一致性 | 25% | 整页截图、区域 diff、bbox、样式、字体、颜色、间距 |
| 结构语义一致性 | 10% | 真实 DOM、控件语义、列表/表单/导航结构、反截图伪页面 |
| 内容数据一致性 | 10% | 可见文案、错误提示、结果字段、分页数字、状态文案 |
| 工程可维护性 | 10% | 可构建运行、project-local 源码组织、样式隔离、评估产物完整 |

## 问题模型

每个问题都会被归入一个维度，并包含：

- `dimension`：六维之一。
- `severity`：`blocker`、`error`、`warning`、`info`。
- `impactScore`：对评分和优先级的影响。
- `confidence`：问题归因置信度。
- `suggestedFix`：可执行修复建议。
- `artifactLinks`：关联截图、diff、交互失败截图或采集 JSON。

问题优先级优先考虑 blocker/error，其次考虑功能、交互、视觉这些核心维度。
