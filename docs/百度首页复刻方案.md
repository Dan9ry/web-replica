# 百度首页复刻方案

> 目标页面：`https://www.baidu.com`  
> 复刻路由：`/replica/baidu`  
> 代码目录：`src/pages/BaiduReplica/`  
> 评估配置：`evaluator/targets/baidu.json`  
> 方案状态：实施前方案，正式编码前必须重新运行原网页采集门禁。

## 1. 基准获取原则

百度首页会随时间、地区、A/B 实验和登录状态变化。复刻前不能只依赖历史截图，也不能让 AI 主观补全页面细节。

百度复刻不是只复刻首页截图。因为题目范围包含“搜索结果文案展示、结果列表、翻页”，所以必须先获取所有真实页面状态，再启动复刻实现。真实状态包括：首页、真实搜索结果页、分页状态，以及空输入行为。任一必需状态获取失败，都必须暂停复刻并汇报，不能自行设计结果页格式。

本方案参考 `web-clone` 的 URL 驱动复刻流程，将百度首页复刻拆成五个闭环阶段：

```text
Pass 1 Extract → Pass 2 Build Plan → Pass 3 Generate Clone → Pass 4 Verify → Pass 5 Iterate
```

改进后的核心原则：

- 先从真实 URL 提取页面结构、样式、交互和视觉基准，再生成代码。
- 使用 accessibility snapshot 和 DOM/computedStyle 提取作为主要结构依据，截图只作为视觉验证依据。
- 每轮验证都输出视觉、功能、结构三部分的复合 verdict。
- 只修复 verdict 中最高优先级问题，不做无关重构。
- 最多迭代 5 轮；若仍未达标，输出最佳结果和剩余差异。

正式复刻前必须执行：

```bash
npm run eval:capture
```

正式复刻前必须成功获取的真实状态：

| 状态 ID | 真实状态 | 触发方式 | 用途 |
| --- | --- | --- | --- |
| `home` | 百度首页初始态 | 打开 `https://www.baidu.com` | 首页布局、搜索框、热榜基准 |
| `results` | 百度搜索结果页 | 输入“微信支付”后按 Enter 或点击搜索 | 顶部小 Logo、结果页搜索框、分类 Tab、结果列表、侧栏基准 |
| `results-page-2` | 搜索结果分页态 | 在真实结果页点击第 2 页或下一页 | 分页样式和列表变化基准 |
| `empty-query` | 空搜索行为 | 空输入提交 | 校验真实站是否提示、保持原页或有其他反馈 |

以上状态的截图、DOM/可访问性摘要、关键样式和触发步骤都要保存。当前实现发现：真实百度点击搜索后可能先进入中间态并提示“请按‘回车’键发起检索”，必须继续按 Enter 获取真正结果页，不能把中间态误当成结果页。

如果出现以下情况，必须停止复刻并汇报：

- Playwright 或 `@chrome` 无法打开真实百度页面。
- 页面进入空白页、错误页、验证码页、访问受限页。
- 截图为空或尺寸异常。
- 标题、URL、正文关键文本与 `https://www.baidu.com` 不匹配。
- 关键搜索区域无法确认。

## 2. web-clone 流程适配

### Pass 1：Extract

目标：采集百度首页的可验证基准，而不是凭截图猜测。

采集内容：

| 类型 | 采集方式 | 用途 |
| --- | --- | --- |
| 页面加载状态 | Playwright / `@chrome` 打开真实 URL | 判断页面是否可访问、是否被拦截 |
| Accessibility snapshot | 浏览器可访问性树 | 识别 nav、main、form、button、link、textbox 等语义结构 |
| 截图 | desktop/mobile 两个视口截图 | 视觉基准、diff 和 SSIM |
| DOM + computedStyle | `window.getComputedStyle` 抽样 | 提取字号、颜色、间距、圆角、布局方式 |
| 交互元素清单 | `button, a, input, textarea, [role=button]` | 建立交互地图 |
| 网络请求模式 | 可选记录 XHR/fetch | 只做参考，不复刻后端接口 |

提取数据必须落盘，便于中断后继续：

```text
reports/latest/state/baidu-web-clone-extraction.json
reports/latest/state/baidu-web-clone-verdicts.json
reports/latest/assets/baidu/home/original-desktop.png
reports/latest/assets/baidu/results/original-desktop.png
reports/latest/assets/baidu/results-page-2/original-desktop.png
reports/latest/assets/baidu/empty-query/original-desktop.png
reports/latest/assets/baidu/home/original-mobile.png
```

提取数据体积控制：

- DOM 树超过 30KB 时，降低递归深度，只保留顶层结构和搜索区域。
- Accessibility snapshot 超过 20KB 时，只摘要 landmark、form、input、button、link。
- 交互元素最多保留 50 个，优先保留可见元素。
- 截图只在基准采集和最终验证时保存，调试单一区域时再额外截图。

### Pass 2：Build Plan

目标：把提取结果转成组件和实现计划。

分析维度：

| 维度 | 百度页关注点 |
| --- | --- |
| 页面区域 | 顶部导航、Logo 搜索主区、文心入口、热榜、右下角悬浮工具 |
| 组件映射 | 每个区域对应 React 组件、props、状态来源 |
| 交互地图 | 搜索提交、Enter、空输入、分页、热榜换一换、focus/hover |
| Design tokens | 蓝紫主色、正文色、弱文本、边框、圆角、间距、字号 |
| 语义结构 | `<nav>`、`<main>`、`<form role="search">`、`<button>`、`<ol>` |

### Pass 3：Generate Clone

目标：按组件计划生成代码。

实现顺序：

1. 先写 mock 数据和类型。
2. 再写 design tokens 与页面骨架。
3. 由外到内实现组件。
4. 接入搜索和分页状态。
5. 补充 hover、focus、active、empty error 状态。
6. 加移动端响应式。

### Pass 4：Verify

目标：每轮验证都输出可比较结果。

验证分三部分：

| 维度 | 检查方式 | 通过线 |
| --- | --- | --- |
| Visual | 原页截图 vs 复刻截图，SSIM/像素差/关键元素位置 | 视觉得分 `>= 85` |
| Functional | Playwright 执行 2-3 个核心交互 | 无失败交互 |
| Structure | 比较 nav/main/form/button/input/list 等 landmark | 主要结构不缺失 |

每轮输出复合 verdict：

```json
{
  "visual": {
    "score": 0,
    "verdict": "revise",
    "differences": [],
    "suggestions": []
  },
  "functional": {
    "tested": 0,
    "passed": 0,
    "failures": []
  },
  "structure": {
    "landmark_match": false,
    "missing": [],
    "extra": []
  },
  "overall_verdict": "revise",
  "priority_fixes": []
}
```

整体通过条件：

- `visual.score >= 85`
- `functional.failures` 为空
- `structure.landmark_match === true`

### Pass 5：Iterate

目标：基于 verdict 做小步修复。

修复优先级：

```text
布局结构 > 核心交互 > 关键间距 > 字体 > 颜色 > 装饰细节
```

迭代规则：

- 每轮只修复 `priority_fixes` 中最高影响的问题。
- 每轮修复后重新运行验证。
- 最多 5 轮。
- 5 轮后仍未通过时，提交当前最佳结果，并在报告中列出剩余差异。

## 3. 当前页面观察

基于本地已保留的参考截图，百度首页当前首屏大致包含：

| 区域 | 内容 |
| --- | --- |
| 顶部导航 | 新闻、hao123、地图、贴吧、视频、图片、网盘、文库、文心、搭子 DuMate、更多 |
| 右上角 | 设置、登录按钮 |
| 主视觉 | 百度 Logo，居中显示 |
| 搜索区 | 大尺寸圆角搜索框、输入占位文案、语音/附件/图片图标、蓝紫色“百度一下”按钮 |
| AI 引导 | “文心 复杂问题就找文心助手，深入思考回答更优”胶囊入口 |
| 热榜区 | 百度热搜标题、换一换、左右两列热搜条目、热/新标识 |
| 右下角悬浮 | 圆形辅助按钮、二维码入口 |

题目要求的复刻范围是：

- 搜索框。
- 搜索按钮点击交互。
- 搜索结果文案展示。
- 结果列表翻页功能。

因此页面视觉上保留首页核心区域和热榜氛围，但实现重点放在搜索行为与结果列表闭环。

## 4. 复刻范围

### 必做范围

| 功能 | 说明 |
| --- | --- |
| 首页搜索框 | 支持输入关键词、清空和 Enter 提交 |
| 搜索按钮 | 点击后触发本地模拟搜索 |
| 搜索结果文案 | 展示“百度为您找到相关结果约 N 个”等本地模拟文案 |
| 结果列表 | 展示标题、摘要、来源、时间 |
| 分页 | 支持上一页、下一页、页码切换 |
| 空输入校验 | 空内容提交时保持首页状态或显示轻提示 |
| 响应式 | 桌面端居中布局，移动端搜索区不溢出 |

### 可做但不阻塞

| 功能 | 处理方式 |
| --- | --- |
| 顶部导航链接 | 仅展示，不跳转真实站点 |
| 设置菜单 | 可展示 hover 状态，不实现完整菜单 |
| 登录按钮 | 仅展示，不实现登录 |
| 热搜换一换 | 使用本地 mock 数据切换一组热搜 |
| 图标按钮 | 作为视觉元素，点击不触发真实上传或语音 |
| 文心入口 | 仅展示，不跳转 |

### 不做范围

- 不调用百度真实搜索接口。
- 不抓取真实搜索结果。
- 不实现真实登录、图片搜索、语音搜索、文件上传。
- 不绕过验证码、风控或访问限制。

## 8. 交互流程

### 搜索流程

1. 用户进入 `/replica/baidu`。
2. 页面展示首页状态。
3. 用户输入关键词，例如“微信支付”。
4. 用户点击“百度一下”或按 Enter。
5. 页面进入短暂 `searching` 状态。
6. 页面展示结果页：
   - 结果摘要文案包含关键词。
   - 结果列表出现。
   - 当前页默认为 1。
7. 用户点击下一页。
8. 当前页变为 2，结果列表更新。

结果页必须参考真实百度结果页，而不是自行设计普通列表页。真实结果页观察到的关键结构：

| 区域 | 真实表现 |
| --- | --- |
| 顶部 | 左侧小百度 Logo，横向搜索框，右侧用户导航 |
| 分类栏 | 文心、网页、图片、资讯、视频、笔记、地图、贴吧、文库、更多、搜索工具 |
| 主结果列 | 左侧约 608px 宽，结果卡片从 `x≈150` 开始，标题约 18px，正文 13px 左右 |
| 侧栏 | 右侧相关搜索、百度热搜、换一换 |
| 特殊卡片 | 官方结果、百科结果、AI 总结、大家还在搜等可能混排 |
| 分页 | 页面底部 1-10 页数字分页 |

如果真实结果页未采集成功，不允许实现结果列表。

### 空输入流程

1. 用户不输入内容。
2. 点击“百度一下”。
3. 不跳转结果页。
4. 输入框展示轻提示，例如“请输入搜索关键词”。
5. 自动评估断言页面仍处于首页模式。

### 热搜换一换

1. 用户点击“换一换”。
2. `hotListGroup` 在本地数组之间切换。
3. 热搜内容更新，不影响搜索框内容。

### 桌面端

| 元素 | 目标 |
| --- | --- |

### 移动端

| 元素 | 目标 |
| --- | --- |
| 顶部导航 | 隐藏大部分链接，仅保留 Logo/登录或简化布局 |

## 10. 样式策略

- 使用 CSS Modules，避免污染其他复刻页面。
- 使用 CSS Variables 记录百度页专用颜色：

```css
:root {
  --baidu-blue: #4e6ef2;
  --baidu-blue-hover: #4662d9;
  --baidu-text: #222222;
  --baidu-muted: #626675;
  --baidu-border: #cfd6ff;
  --baidu-bg-soft: #f5f6fb;
}
```

- 视觉优先匹配当前参考截图，但保留合理容错：
  - 搜索框宽度误差不超过 16px。
  - 搜索按钮位置误差不超过 12px。
  - 颜色 RGB 平均差不超过 15。
  - 首页首屏 SSIM 目标不低于 0.88。

## 11. 自动评估设计

### 功能一致性用例

| 用例 | 断言 |
| --- | --- |
| 初始首页 | 搜索框、搜索按钮、Logo、热榜可见 |
| 输入关键词 | 输入框值等于关键词 |
| 点击搜索 | 结果摘要和结果列表出现 |
| Enter 搜索 | 与点击按钮效果一致 |
| 空输入 | 不进入结果页，显示提示或保持首页状态 |
| 下一页 | 当前页码更新，结果列表变化 |
| 上一页 | 当前页码回退 |
| 换一换 | 热榜内容变化 |

### 交互一致性用例

| 交互 | 断言 |
| --- | --- |
| 输入框 focus | 边框高亮 |
| 搜索按钮 hover | 背景色变深 |
| 页码点击 | active 样式变化 |
| Tab 顺序 | 输入框 → 图标按钮 → 搜索按钮 |
| 移动端 | 搜索区域不溢出 |

### 视觉一致性评估

使用当前评估框架扩展以下检查：

| 指标 | 阈值 |
| --- | --- |
| 首页截图 SSIM | `>= 0.88` |
| 结果页截图 SSIM | `>= 0.82` |
| 搜索框位置误差 | `<= 16px` |
| 搜索按钮尺寸误差 | `<= 12px` |
| Logo 中心点误差 | `<= 20px` |
| 热榜区域宽度误差 | `<= 24px` |

当前 `evaluator/targets/baidu.json` 已包含：

### 结构一致性评估

引入 web-clone 的 landmark 对比思路，不要求 DOM 完全一致，但要求主要语义结构完整。

| 原网页结构 | 复刻要求 |
| --- | --- |
| 顶部导航 | 至少存在一个 `<nav>` 或等价导航区域 |
| 搜索表单 | 使用 `<form role="search">` 或可识别的搜索表单结构 |
| 搜索输入 | 使用文本输入控件，具备 `aria-label` 或可访问名称 |
| 搜索按钮 | 使用真实 `<button>`，可通过键盘触发 |
| 热榜列表 | 使用列表结构，例如 `<ol>` |
| 结果列表 | 搜索后使用列表语义组织结果 |
| 分页 | 使用按钮或链接，当前页有明确状态 |

### 交互抽样验证

每轮至少抽测 3 个交互：

| 交互 | 通过条件 |
| --- | --- |
| 输入关键词并点击搜索 | 出现结果摘要和结果列表 |
| 按 Enter 搜索 | 与点击按钮结果一致 |
| 点击下一页 | 页码和结果内容变化 |

如果其中任意一个失败，`functional.failures` 不为空，整体 verdict 为 `revise`。

## 12. 错误处理策略

| 问题 | 处理 |
| --- | --- |
| 浏览器工具不可用 | 停止复刻，提示需要恢复 Playwright 或 `@chrome` 能力 |
| 原页面加载失败 | 输出 URL、状态码、最终 URL，不生成基准 |
| `browser_evaluate` 返回空 | 等待后重试一次；仍为空则停止并汇报 |
| 页面被验证码/风控拦截 | 停止，不绕过，不猜测 |
| 提取数据过大 | 降低 DOM 深度，保留顶层结构和搜索区域 |
| 视觉得分 3 轮低于阈值 | 汇报主要差异，优先修复布局与搜索区 |
| 5 轮仍未通过 | 输出最佳结果和剩余问题 |

## 16. 验收标准

- `/replica/baidu` 本地可访问。
- 首页搜索框、搜索按钮、结果文案、结果列表和分页可用。
- 不访问真实百度搜索接口。
- 空输入、Enter 搜索、按钮搜索、分页切换均有自动化测试。
- 评估报告中百度页面有功能、交互和视觉评分。
- 每轮评估输出包含 visual、functional、structure 的复合 verdict。
- 视觉得分目标不低于 85。
- 功能抽样验证无失败项。
- 主要 landmark 结构匹配。
- 原网页采集失败时，不生成百度一致性总分。
- AI Prompt 和关键修改过程被记录。
