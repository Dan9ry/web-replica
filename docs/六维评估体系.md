# 六维评估体系

更新时间：2026-05-30

本文说明 `web-replica` 项目当前使用的网页复刻六维评估体系。它的目标不是评价原网站本身，而是判断复刻页面是否在用户指定的复刻范围内，尽可能还原真实网站的桌面端视觉、结构、内容、功能与交互。

## 1. 评估目标

评估体系回答三个问题：

1. 复刻页面是否实现了用户要求的核心功能。
2. 复刻页面是否在视觉、结构、内容和交互上接近真实页面。
3. 复刻产物是否是可运行、可维护、可继续迭代的真实网页。

最终复刻产物必须是由 HTML、CSS、JavaScript 等源码实现的可交互网页。禁止把整页截图、大区域截图、canvas 静态画面或图片热区当作最终网页。

## 2. 评估输入

每个复刻项目都有独立目录：

```text
projects/{target-id}/
```

评估时指定当前项目配置：

```bash
EVAL_TARGET_CONFIG=projects/{target-id}/config/target.json npm run eval
```

评估器主要读取：

| 输入 | 作用 |
| --- | --- |
| `config/target.json` | 定义复刻范围、页面状态、关键 selector、交互用例和评分目标 |
| `baselines/{state-id}/original-*.png` | 原站真实截图基线 |
| `baselines/{state-id}/original-dom.json` | 原站可见 DOM、文本、布局和样式摘要 |
| 复刻页面 URL | 用于采集复刻页面截图、DOM 和交互结果 |

评估结果保存在对应项目目录下：

```text
projects/{target-id}/evaluation/latest/
```

报告会标明本次评估使用的是实时原站采集基线，还是用户提供的截图基线。

## 3. 核心原则

1. 先有可信原站基线，再做复刻和评分。
2. 评估阶段不重新交互式采集原站，只使用前置阶段已确认的截图、DOM 摘要、关键 selector 和文本基线。
3. 原站基线缺失、错误页、验证码页、安全验证页、空白页或关键状态缺失时，不能主观猜测，应中断采集或标记门禁失败。
4. 评估只针对当前指定项目，不默认评估所有项目。
5. 复刻范围不等于功能范围：用户没有要求实现功能的可见区域，仍然应该影响视觉一致性，但不进入功能和交互用例。
6. 登录、支付、账号、安全验证类页面只采集可见 UI 证据，不采集隐藏 input、cookie、localStorage、sessionStorage 或 token 字段。

## 4. 硬门禁

硬门禁优先于评分。硬门禁失败时，报告仍可生成，但结论不能视为通过。

| 门禁 | 通过条件 |
| --- | --- |
| 原站基线可信 | required 状态都有真实截图、DOM 摘要、可信 URL/title、核心文本和关键 selector |
| 复刻页可运行 | 本地或线上 URL 可访问，页面不是空白页 |
| required 状态可达 | 用户要求的每个核心状态都能进入和采集 |
| 禁止截图伪页面 | 主体 UI 不是由整页截图、大图背景、canvas 或图片热区伪装 |
| 核心交互无阻断错误 | 核心流程没有白屏、死循环、未捕获异常或无法继续的错误 |
| 评估可复现 | 配置、基线、截图、交互记录和报告产物完整 |

## 5. 六维总分

总分采用 100 分制，只评估桌面端。

```text
总分 =
  功能正确性       * 25%
+ 交互流程一致性   * 20%
+ 视觉一致性       * 25%
+ 结构语义一致性   * 10%
+ 内容数据一致性   * 10%
+ 工程可维护性     * 10%
```

每个维度满分 100。报告会分别展示六维分数、总分、问题列表和低分项修改建议。

## 6. 默认验收阈值

当前 skill 默认验收阈值如下：

| 指标 | 阈值 |
| --- | ---: |
| 总分 | 85 |
| 功能正确性 | 85 |
| 交互流程一致性 | 85 |
| 视觉一致性 | 60 |
| 结构语义一致性 | 80 |
| 内容数据一致性 | 80 |
| 工程可维护性 | 80 |

如果用户要求更高标准，则使用用户指定的更高阈值。除非用户明确接受更低目标，否则不主动降低默认阈值。

## 7. 功能正确性

功能正确性评估复刻页面是否完成了用户明确要求的核心功能，例如搜索、登录表单、翻页、输入校验、结果展示、模式切换等。

### 整体公式

```text
功能正确性 =
  状态完整性       * 25%
+ 交互任务通过率   * 35%
+ 内容反馈正确性   * 20%
+ 边界情况处理     * 20%
```

### 分项说明

| 分项 | 含义 | 计算方式说明 |
| --- | --- | --- |
| 状态完整性 | 用户要求的页面状态是否都被实现，例如首页、搜索结果页、无结果页、错误提示页 | 已成功采集并匹配的复刻状态数 / 需求状态数 * 100 |
| 交互任务通过率 | 核心功能操作是否能正确完成，例如输入关键词后搜索、点击下一页、提交表单 | 通过的交互任务权重 / 全部交互任务权重 * 100 |
| 内容反馈正确性 | 操作之后是否出现正确反馈，例如结果列表、提示文案、状态变化 | 已正确展示的反馈内容数量 / 需求反馈内容数量 * 100 |
| 边界情况处理 | 空输入、非法输入、无结果、错误态等是否有合理响应 | 通过的边界用例权重 / 全部边界用例权重 * 100；没有边界用例时沿用交互任务通过率 |

### 判分说明

用户明确要求的功能会被设置为较高权重。比如搜索页面中，搜索输入、按钮点击、结果展示和翻页属于核心功能；顶部账号入口如果用户没有要求实现登录功能，则不进入功能正确性评分，但仍会影响视觉一致性。

具体实现中，评估器会读取项目配置中的页面状态和交互脚本，运行复刻页面后按权重汇总通过情况。

## 8. 交互流程一致性

交互流程一致性评估复刻页面的操作过程是否接近真实网站，包括输入、点击、跳转、实时反馈、分页、hover、focus、blur、模式切换等。

### 整体公式

```text
交互流程一致性 =
  通过的交互任务权重总和 / 全部交互任务权重总和 * 100
```

每个交互任务由“操作能否执行、操作后状态是否正确、断言结果是否通过”共同判定。用户明确要求的核心交互会配置更高权重，因此核心交互失败会比辅助交互失败扣分更多。

如果当前项目没有配置交互脚本，评估器无法真实执行点击、输入、翻页等流程。当前实现会临时参考“需求状态是否可达”估算交互分，但报告中应明确说明该分数可信度较低，正式验收前应补充交互脚本。

### 分项说明

| 分项 | 含义 | 计算方式说明 |
| --- | --- | --- |
| 交互任务通过率 | 自动化脚本执行用户要求的操作后，是否达到预期状态 | 按任务权重统计通过率：通过任务权重总和 / 全部任务权重总和 * 100 |
| 操作执行结果 | 输入、点击、按键、hover、focus 等动作是否能正常执行 | 单个任务中任一关键操作无法执行，该任务通常判为失败 |
| 状态变化结果 | 操作后页面是否进入预期状态，例如进入搜索结果页、显示下一页结果 | 状态变化符合预期时，该任务的状态断言通过；不符合则该任务扣分或失败 |
| 结果断言结果 | 操作后目标文本、URL、元素可见性、输入值等是否符合预期 | 断言全部通过时任务通过；关键断言失败时任务失败 |
| 关键交互权重 | 用户明确要求的核心交互比辅助交互更重要 | 核心任务在总权重中占比更高，因此失败时对总分影响更大 |

### 判分说明

交互评分更关注“用户是否能按真实流程完成任务”。动画、过渡和实时反馈如果是目标网站的重要体验，也应作为交互用例或状态变化记录下来。

具体实现中，评估器通过 Playwright 执行配置里的交互任务，并根据断言结果计算通过率。

## 9. 视觉一致性

视觉一致性评估复刻页面看起来是否像真实页面。它关注整页观感，也关注用户需求相关的关键区域，例如搜索框、结果列表、登录表单、页头、页尾等。

### 整体公式

配置了关键区域时：

```text
视觉一致性 =
  关键区域相似度   * 45%
+ 整页截图相似度   * 15%
+ 布局一致性       * 20%
+ 样式一致性       * 15%
+ 基础可用奖励分
```

没有配置关键区域时：

```text
视觉一致性 =
  整页截图相似度   * 75%
+ 布局一致性       * 15%
+ 样式一致性       * 10%
```

### 分项说明

| 分项 | 含义 | 计算方式说明 |
| --- | --- | --- |
| 整页截图相似度 | 原站整页截图和复刻页面整页截图整体是否接近 | 综合像素差异和结构相似度，差异越小、结构越接近，分数越高 |
| 关键区域相似度 | 搜索框、结果列表、登录表单等重点区域是否接近 | 对每个关键区域分别比较截图、结构相似度、位置大小和样式，再求平均 |
| 布局一致性 | 关键元素的位置、大小、间距是否接近真实页面 | 对关键 selector 的位置和尺寸进行比较，偏差越小得分越高 |
| 样式一致性 | 字体、字号、颜色、背景、圆角等是否接近真实页面 | 对关键 selector 的主要 CSS 样式进行比较，差异越小得分越高 |
| 基础可用奖励分 | 页面正常渲染、关键区域可见、没有明显空白或渲染失败 | 在配置关键区域时，页面基础可用会提供少量稳定分 |

### 判分说明

视觉一致性不要求像素级完全一致，但关键区域应明显接近真实页面。用户未要求实现功能的可见区域，例如顶部账户区、应用入口、页脚链接，也会影响视觉一致性，因为它们属于页面外观的一部分。

具体实现中，评估器会生成原站和复刻页面截图，并结合像素差异、SSIM、元素盒模型和关键样式计算视觉分。

## 10. 结构语义一致性

结构语义一致性评估复刻页面是否由真实 DOM 和真实控件组成，而不是用截图伪装。它允许 DOM 结构存在合理差异，不要求和原站完全一致，但用户要求的核心结构必须准确。

### 整体公式

```text
结构语义一致性 =
  关键结构覆盖率   * 80%
+ 页面基础结构     * 10%
+ 控件语义正确性   * 4%
+ 可聚焦/可操作性  * 3%
+ 文本真实性       * 3%
```

### 分项说明

| 分项 | 含义 | 计算方式说明 |
| --- | --- | --- |
| 关键结构覆盖率 | 用户复刻范围内的关键 DOM 是否存在，例如 input、button、form、结果列表、分页控件 | 复刻页面命中的关键 selector 加权数量 / 需求关键 selector 加权数量 * 100 |
| 页面基础结构 | 页面是否具备合理的 header、main、footer、nav、form、list 等基础结构 | 复刻页面基础结构数量与原站基线接近程度，越接近分数越高 |
| 控件语义正确性 | 搜索框是否是 input，按钮是否是 button，链接是否是 a，表单是否是 form | 语义正确的关键控件数量 / 关键控件总数量 * 100 |
| 可聚焦/可操作性 | 输入框、按钮、链接等是否可以被键盘或脚本聚焦和操作 | 可聚焦或可操作控件数量 / 关键交互控件数量 * 100 |
| 文本真实性 | 页面文字是否是真实 DOM 文本，而不是画在图片里 | 存在真实可见文本则得高分；主要文字缺失或被图片替代则扣分 |

### 判分说明

结构评分按照复刻需求加权。用户明确要求的核心结构权重更高，例如搜索输入框、搜索按钮、结果列表和分页控件；只承担视觉作用的链接或账户入口权重较低，但仍应作为可见结构存在。

具体实现中，评估器根据项目配置的关键 selector、原站 DOM 摘要和复刻 DOM 摘要计算结构覆盖情况，并检查是否存在截图伪页面风险。

## 11. 内容数据一致性

内容数据一致性评估复刻页面显示的文字、结果、提示和状态内容是否符合真实页面和用户需求。

### 整体公式

```text
内容数据一致性 =
  关键文案覆盖率   * 55%
+ 文本规模相似度   * 25%
+ 状态内容完整性   * 20%
```

### 分项说明

| 分项 | 含义 | 计算方式说明 |
| --- | --- | --- |
| 关键文案覆盖率 | 用户要求或配置中指定的标题、按钮、提示、结果文案是否出现 | 已出现的关键文案数量 / 期望关键文案数量 * 100 |
| 文本规模相似度 | 页面文本量是否与真实页面大致接近，避免内容明显缺失 | 复刻页面文本长度与原站文本长度越接近，分数越高 |
| 状态内容完整性 | 不同页面状态是否都有对应内容，例如首页、搜索结果页、错误提示页 | 有完整内容的状态数量 / 需求状态数量 * 100 |

### 判分说明

内容评分主要看“该有的文字和状态内容是否出现”。对于搜索结果页，如果用户要求展示结果文案和列表，结果标题、摘要、分页数字等都应进入内容评估；如果用户只要求某些字段，未要求字段不会按功能缺失扣分，但明显影响页面观感的内容仍会影响视觉评分。

具体实现中，评估器会检查配置中要求出现的文本，并比较原站与复刻页面在不同状态下的可见文本摘要。

## 12. 工程可维护性

工程可维护性评估复刻产物是否能稳定运行、独立维护和长期迭代。

### 整体公式

```text
工程可维护性 =
  运行稳定性       * 40%
+ 项目结构规范性   * 30%
+ 证据完整性       * 30%
```

### 分项说明

| 分项 | 含义 | 计算方式说明 |
| --- | --- | --- |
| 运行稳定性 | 复刻页面是否能正常启动、加载、截图和执行交互 | 页面可访问且采集无错误得高分；白屏、加载失败、运行异常会扣分 |
| 项目结构规范性 | 复刻源码、配置、基线和报告是否放在对应 project 目录下 | 符合 `projects/{target-id}` 目录规范得高分，不规范则扣分 |
| 证据完整性 | 是否保留真实截图、DOM 摘要、配置、评估报告和问题记录 | 证据文件越完整，分数越高 |

### 判分说明

工程可维护性不只看页面能不能“看起来能跑”，还看它是否便于后续维护和复评。复刻产物应该能在本地或线上独立运行，并且源码应清晰可读，不应依赖一次性脚本或不可维护的硬编码堆叠。

具体实现中，评估器会检查页面运行结果、项目目录位置和评估证据文件完整性。

## 13. 评分等级

| 分数范围 | 等级 | 含义 |
| --- | --- | --- |
| 90 - 100 | 高一致 | 复刻质量较高，可作为较成熟成果提交 |
| 80 - 89 | 基本一致 | 主要目标已达成，仍有局部可优化项 |
| 70 - 79 | 可运行但需修复 | 页面可用，但低分项明显，需要继续迭代 |
| 70 以下 | 不建议提交验收 | 关键一致性不足，应先修复核心问题 |

## 14. 低分项如何处理

评估报告会对低分维度给出修改建议。一般处理顺序如下：

1. 先修复硬门禁问题，例如原站基线缺失、复刻页不可访问、截图伪页面风险。
2. 再修复功能和交互问题，因为它们直接决定用户要求是否实现。
3. 然后修复结构、内容和视觉问题，使页面更接近真实网站。
4. 每轮只重新评估当前 project，最多迭代三轮，直到达到目标分数或明确记录未达标原因。

## 15. 实现位置

本文只说明评分体系和判分方式。具体代码实现位于：

| 文件 | 作用 |
| --- | --- |
| `evaluator/core/evaluationMetrics.ts` | 计算六维指标、分项分数和总分 |
| `evaluator/core/scoring.ts` | 定义总分权重和评分等级 |
| `evaluator/collectors/capture-all.ts` | 采集复刻页面截图、DOM、样式和交互结果 |
| `evaluator/reporters/` | 生成 HTML/Markdown 评估报告 |

实现细节会基于项目配置、原站基线和复刻页面运行结果自动计算，不需要用户手动打分。
