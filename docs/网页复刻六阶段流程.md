# 网页复刻六阶段流程

本文档参考 `ppt-master` 的串行流水线思想，将网页复刻流程固化为六个阶段。每个阶段必须以前一阶段产物为输入；遇到阻断条件时停止推进，不允许主观猜测真实页面。

核心流水线：

```text
复刻请求 -> 项目初始化 -> 真实素材采集 -> 复刻策略确认 -> 复刻实现 -> 评估迭代交付
```

## 全局执行规则

1. 串行执行：六个阶段按顺序推进，前一阶段未完成不得进入下一阶段。
2. 门禁优先：每阶段必须产出明确文件或记录，缺失则不得继续。
3. 阻断即暂停：真实网站无法访问、出现验证码/安全验证、截图缺失、页面状态无法确认时，立即暂停并请求用户介入。
4. 证据留痕：每个页面状态都要保存真实截图、触发步骤、DOM/样式摘要或用户提供截图来源。
5. 先规格后实现：用户确认复刻方案前，不开始写页面代码。
6. 只评估三项：功能一致性、交互一致性、视觉一致性。
7. 项目隔离：每个复刻目标必须创建独立 `projects/{target-id}/` 目录，任务单、规格、素材、日志、评估报告和配置副本都集中放在该目录下。
8. 评估方式前置：截图评估、普通自动评估或交互辅助评估必须在复刻方案阶段确定；阶段六只执行已确认方式，除非发生阻断并记录变更。

## 阶段一：复刻请求解析

目标：把用户的自然语言请求整理成可确认的复刻任务单。

### 输入

用户至少需要提供以下信息之一：

- 网站地址。
- 网站截图。
- 网站地址 + 截图的混合素材。

同时需要明确或由我补全以下范围：

- 要复刻哪些页面。
- 每个页面有哪些状态。
- 哪些功能需要复刻。
- 哪些功能明确不复刻。

### 处理动作

1. 判断素材来源类型：
   - `url`：由我访问真实网站并采集截图、DOM 和交互状态。
   - `screenshot`：基于用户截图复刻，不能推断截图之外的页面状态。
   - `mixed`：优先使用真实网站采集，用户截图作为补充证据。
2. 拆分页面范围：
   - 首页。
   - 列表页 / 搜索结果页。
   - 详情页 / 登录页 / 表单页。
   - 弹窗、错误页、空状态、分页页等特殊状态。
3. 拆分功能范围：
   - 输入、点击、提交、分页、筛选、Tab 切换、表单校验等。
   - 真实接口、真实登录、真实支付、真实上传等默认列为不做范围，除非用户明确要求。
4. 明确桌面端视口：
   - 默认只做 PC 端。
   - 默认采集视口为 `1365 x 768`，可按目标网站调整。

### 输出

创建复刻任务单：

```text
projects/{target-id}/request.md
```

任务单必须包含：

```markdown
# {网站名}复刻任务单

## 1. 素材来源

| 字段 | 内容 |
| --- | --- |
| 来源类型 | URL / 截图 / 混合 |
| URL |  |
| 用户截图 |  |
| 采集方式 | 自动采集 / 人工验证后采集 / 用户截图 |

## 2. 页面范围

| 页面 | URL / 截图 | 是否必做 | 说明 |
| --- | --- | --- | --- |

## 3. 状态范围

| 页面 | 状态 | 触发方式 | 是否必需 | 需要真实截图 |
| --- | --- | --- | --- | --- |

## 4. 功能范围

| 功能 | 是否复刻 | 复刻方式 | 说明 |
| --- | --- | --- | --- |

## 5. 不做范围

- 

## 6. 评估目标

| 指标 | 目标分 |
| --- | ---: |
| 功能一致性 | >= 90 |
| 交互一致性 | >= 85 |
| 视觉一致性 | >= 80 |
| 总分 | >= 85 |

## 7. 评估方式

| 字段 | 内容 |
| --- | --- |
| 评估模式 | 普通自动评估 / 交互辅助评估 / 截图来源评估 |
| 原站证据 | 实时原站采集 / 截图基线 / 用户截图 |
| 验证处理 | 自动降级到基线 / 暂停等待用户验证 / 缺失即中断 |
| 适用原因 |  |

## 8. 阻断规则

- URL 无法访问时暂停。
- 出现验证码、安全验证、登录墙时暂停。
- 必需状态截图缺失时暂停。
- 无法确认真实页面时暂停。
```

### 阶段门禁

- 任务单已创建。
- 页面范围、状态范围、功能范围、不做范围已明确。
- 推荐评估方式已写入任务单。
- 用户确认任务单，或明确授权我按推荐范围继续。

## 阶段二：复刻项目初始化

目标：为该目标网站创建独立 project 文件夹，使该复刻目标的请求、素材、规格、日志、评估报告和配置副本都集中管理。

### 输入

- 已确认的复刻任务单。
- 目标网站名称和 `target-id`。

### 处理动作

1. 创建 project 根目录：

```text
projects/{target-id}/
```

2. 在 project 内创建标准子目录：

```text
projects/{target-id}/
├── request.md
├── spec.md
├── logs/
│   ├── ai-log.md
│   ├── decisions.md
│   └── blockers.md
├── prompts/
│   └── replica-prompts.md
├── sources/
│   ├── user-screenshots/
│   └── urls.md
├── baselines/
│   └── {state-id}/
├── captures/
├── evaluation/
│   ├── latest/
│   └── history/
└── config/
    └── target.json
```

3. 创建或预留页面实现目录：

```text
src/pages/{TargetReplica}/
```

4. 创建评估配置草稿，并在 project 中保留副本：

```text
evaluator/targets/{target-id}.json
projects/{target-id}/config/target.json
```

### 输出

本阶段应形成以下文件或占位：

```text
projects/{target-id}/request.md
projects/{target-id}/spec.md
projects/{target-id}/logs/ai-log.md
projects/{target-id}/logs/decisions.md
projects/{target-id}/logs/blockers.md
projects/{target-id}/prompts/replica-prompts.md
projects/{target-id}/sources/
projects/{target-id}/baselines/
projects/{target-id}/evaluation/
projects/{target-id}/config/target.json
evaluator/targets/{target-id}.json
```

### Project 文件夹职责

| 路径 | 内容 | 说明 |
| --- | --- | --- |
| `request.md` | 用户请求解析后的任务单 | 记录素材来源、页面范围、状态范围、功能范围、不做范围和评估目标 |
| `spec.md` | 用户确认后的复刻规格 | 记录页面设计、状态设计、组件拆分、功能设计和视觉策略 |
| `logs/ai-log.md` | AI 使用日志 | 记录关键 Prompt、AI 输出、人工确认、评估迭代和 commit 对应关系 |
| `logs/decisions.md` | 决策日志 | 记录组件拆分、路由设计、状态模拟、接口替代等实现决策 |
| `logs/blockers.md` | 阻断日志 | 记录验证码、安全验证、访问失败、用户介入和截图降级原因 |
| `prompts/replica-prompts.md` | Prompt 留痕 | 保存用于生成、修复、评估的关键 Prompt |
| `sources/user-screenshots/` | 用户提供截图 | 按页面和状态命名，不能混放未标注截图 |
| `sources/urls.md` | URL 素材来源 | 记录原站入口、状态 URL 和访问说明 |
| `baselines/{state-id}/` | 真实状态基线 | 保存状态截图、DOM、样式和采集记录 |
| `captures/` | 临时采集产物 | 保存当轮浏览器采集结果，可用于排查 |
| `evaluation/latest/` | 最新评估报告副本 | 保存本目标最近一次报告、diff 图和摘要 |
| `evaluation/history/` | 历史评估记录 | 每轮评估按时间归档 |
| `config/target.json` | 评估配置副本 | 与 `evaluator/targets/{target-id}.json` 保持同步，便于 project 自包含 |

### 阶段门禁

- 目录结构存在。
- 评估目标 ID 唯一。
- 页面路由命名确定。
- 后续真实截图、采集记录、评估报告和 AI 日志都能在 `projects/{target-id}/` 中找到。

## 阶段三：真实素材采集与状态基线

目标：在写代码前获取所有必需真实页面状态，形成复刻和评估的唯一事实来源。

### 输入

- 已确认的任务单。
- 网站 URL 或用户截图。
- 页面状态清单。

### 处理动作

#### URL 来源

1. 使用浏览器打开真实网站。
2. 逐个状态执行触发步骤。
3. 对每个状态采集：
   - 页面顶部截图。
   - 页面主体截图。
   - 页面底部截图。
   - 完整视口截图。
   - 关键 DOM 结构。
   - 关键元素样式。
   - 可点击 / 可输入元素清单。
4. 如果页面需要滚动，必须覆盖页面头部、中部和底部，不能只截首屏。
5. 如果出现验证码、安全验证、登录墙或访问异常，立即暂停，请用户介入。

#### 截图来源

1. 整理用户提供的截图。
2. 标注每张截图对应的页面和状态。
3. 检查截图是否覆盖：
   - 页面头部。
   - 页面主体。
   - 页面底部。
   - 弹窗、错误态、空状态等特殊状态。
4. 若关键状态截图缺失，向用户列出缺失项并暂停。

### 输出

每个状态都应落盘：

```text
projects/{target-id}/baselines/{state-id}/original-desktop.png
projects/{target-id}/baselines/{state-id}/original-top.png
projects/{target-id}/baselines/{state-id}/original-middle.png
projects/{target-id}/baselines/{state-id}/original-bottom.png
projects/{target-id}/baselines/{state-id}/original-dom.json
projects/{target-id}/baselines/{state-id}/capture-notes.md
```

`capture-notes.md` 记录：

```markdown
# {state-id} 真实状态采集记录

## 触发步骤

1. 
2. 

## 截图文件

- 顶部：
- 中部：
- 底部：
- 视口：

## 关键元素

| 元素 | selector / 描述 | 作用 |
| --- | --- | --- |

## 采集可信性

- 状态是否确认正确：
- 是否经过人工验证：
- 是否使用截图降级：
- 异常说明：
```

### 阶段门禁

- 每个必需状态都有真实截图。
- 页面头部、中部、底部均有素材覆盖。
- URL 来源页面已通过可信性门禁。
- 截图来源页面已明确页面和状态归属。
- 缺失状态必须补采，不能猜测。

## 阶段四：复刻策略与规格确认

目标：基于真实素材生成复刻方案，并让用户确认后锁定实现规格。

### 输入

- 复刻任务单。
- 所有真实状态截图。
- DOM / 样式 / 交互采集记录。

### 处理动作

输出一组“复刻确认项”，等待用户确认：

| 确认项 | 内容 |
| --- | --- |
| 页面范围 | 本次实际复刻哪些页面 |
| 状态范围 | 每个页面必须覆盖哪些状态 |
| 功能范围 | 哪些交互必须可运行 |
| 不做范围 | 哪些真实能力只做视觉或本地模拟 |
| 视觉重点 | 头部、主体、底部、弹窗、列表、表单等还原重点 |
| 组件拆分 | 页面将拆成哪些组件 |
| 路由设计 | 本地访问路径 |
| 评估目标 | 功能、交互、视觉阈值 |
| 评估方式 | 普通自动评估 / 交互辅助评估 / 截图来源评估，以及验证失败时的处理方式 |

用户确认后，写入规格文件：

```text
projects/{target-id}/spec.md
```

规格文件必须包含：

```markdown
# {网站名}复刻规格

## 1. 页面与状态设计

| 页面 | 状态 | 设计说明 | 对应真实截图 |
| --- | --- | --- | --- |

## 2. 组件设计

| 组件 | 负责区域 | 状态变化 | 参考截图 |
| --- | --- | --- | --- |

## 3. 功能设计

| 功能 | 触发方式 | 复刻行为 | 是否调用真实接口 |
| --- | --- | --- | --- |

## 4. 视觉设计

| 区域 | 还原要点 |
| --- | --- |
| 页面头部 |  |
| 页面主体 |  |
| 页面底部 |  |
| 弹窗/浮层 |  |

## 5. 评估配置设计

| 状态 | selector | 交互步骤 | 截图基准 |
| --- | --- | --- | --- |

## 6. 评估方式锁定

| 字段 | 内容 |
| --- | --- |
| 评估模式 | 普通自动评估 / 交互辅助评估 / 截图来源评估 |
| 原站证据来源 | 实时原站采集 / 截图基线 / 用户截图 |
| 验证失败处理 | 自动降级到基线 / 暂停等待用户验证 / 缺失即中断 |
| 是否允许变更 | 默认不允许；如需变更必须写入 `logs/decisions.md` 和 `logs/blockers.md` |
```

同时更新机器可读配置：

```text
evaluator/targets/{target-id}.json
projects/{target-id}/config/target.json
```

### 阶段门禁

- 用户已确认复刻策略。
- 每个状态都有设计方案和真实截图映射。
- 功能、不做范围、评估目标均已锁定。
- 评估方式已锁定。
- 评估配置能覆盖所有必需状态。

## 阶段五：复刻实现

目标：根据锁定规格实现复刻页面，但不把评估修复混在本阶段。实现阶段只负责把页面、状态和本地交互做出来，并保证本地可运行。

### 输入

- 已确认的复刻规格。
- 真实素材基线。
- 评估配置。

### 处理动作

1. 先实现页面骨架和路由。
2. 按状态逐个实现：
   - 初始态。
   - 输入态。
   - 结果态。
   - 分页态。
   - 空状态 / 错误态。
   - 弹窗 / 浮层。
3. 每实现一个状态，先做本地自检：
   - 页面能打开。
   - 必需元素存在。
   - 输入、点击、分页、弹窗等本地行为可触发。
   - 页面头部、主体、底部都已按规格实现。
4. 记录实现过程：
   - 使用过的关键 Prompt。
   - AI 生成代码后的人工修改。
   - 组件拆分和状态模拟决策。
   - 当前 commit。

### 输出

```text
src/pages/{TargetReplica}/
evaluator/targets/{target-id}.json
projects/{target-id}/config/target.json
projects/{target-id}/logs/ai-log.md
projects/{target-id}/logs/decisions.md
projects/{target-id}/prompts/replica-prompts.md
```

### 阶段门禁

- 本地页面可访问。
- 所有必需状态可触发。
- 页面头部、主体、底部均已实现。
- 真实接口、真实登录、真实支付等不做范围没有被误实现。
- `projects/{target-id}/logs/ai-log.md` 已记录关键 Prompt、AI 输出、人工修改和 commit。

## 阶段六：评估迭代与交付

目标：对已实现页面运行一致性评估，根据报告逐轮修复，直到达到任务单目标或明确记录未达标原因。

### 输入

- 已实现的复刻页面。
- 已确认的复刻规格。
- 真实素材基线。
- 评估配置。

### 处理动作

1. 明确评估目标：
   - 评估永远只针对当前 `projects/{target-id}` 复刻项目。
   - 必须指定 `EVAL_TARGET={target-id}` 或使用当前项目对应脚本。
   - 不提供“评估全部”选项，未实现页面不得被纳入本轮评估。
2. 按阶段四锁定的评估方式执行：
   - `普通自动评估`：实时采集原站；遇验证码、安全验证或 AI 校验时，按规格决定是否降级到最近一次成功截图/DOM 基线。
   - `交互辅助评估`：打开可见浏览器；遇验证时暂停流程，等待用户完成验证后继续。
   - `截图来源评估`：以用户截图或已确认截图基线作为原站证据，不再实时推断截图之外的状态。
   - 如果实际情况要求临时改变评估方式，必须暂停并记录到 `logs/decisions.md` 和 `logs/blockers.md`。
3. 运行三项一致性评估：
   - 功能一致性。
   - 交互一致性。
   - 视觉一致性。
4. 生成评估报告：
   - 总分和等级。
   - 三项分数。
   - 评估方式：普通自动评估、交互辅助评估或截图来源评估。
   - 原站证据：实时原站采集、截图/DOM 基线或用户截图。
   - 视觉 diff 图。
   - 问题列表。
   - 低分项修改建议。
5. 按报告迭代：
   - 功能低分：补齐缺失状态或关键功能。
   - 交互低分：修复点击、输入、提交、分页、状态切换。
   - 视觉低分：查看 diff 图，修复布局、尺寸、字体、颜色、间距。
6. 归档报告和记录：
   - 最新报告复制到 `projects/{target-id}/evaluation/latest/`。
   - 每轮报告按时间复制到 `projects/{target-id}/evaluation/history/`。
   - 在 `logs/ai-log.md` 记录每轮分数、问题、修复和 commit。

### 输出

```text
projects/{target-id}/evaluation/latest/report.md
projects/{target-id}/evaluation/latest/index.html
projects/{target-id}/evaluation/latest/summary.json
projects/{target-id}/evaluation/history/{timestamp}/
projects/{target-id}/logs/ai-log.md
```

### 阶段门禁

- 评估报告只包含功能、交互、视觉三项。
- 报告标明阶段四锁定的评估方式和实际原站证据来源。
- 低分项必须有明确修改建议。
- 只包含当前复刻项目的评估结果。
- 总分达到任务单目标，或明确记录未达标原因和下一步修复建议。

## 六阶段产物总览

| 阶段 | 关键产物 | 是否阻断 |
